# Ad-Serving Architecture — Ranked Ideas

**Grounding.** Read against the actual applied state, not just the design docs: `docs/growth/ARCHITECTURE.md`, `SCALING_COST.md`, `DATA_MODEL.md`, `INAPP_ADS.md`, plus `supabase/migrations/20260712000000_growth_phase0_featured.sql`, `20260712010000_growth_ui_rpcs.sql`, `20260712030000_growth_admin_controls.sql`, `src/lib/api/growth.ts`, `src/pages/Explore.tsx`, `src/components/growth/FeaturedCard.tsx`, `src/lib/queryClient.ts`, and `docs/ops/RATE_LIMITING.md` for house idiom. Two facts drive most of this list: (1) the owner pivoted campaigns from email to **in-app only** (`docs/growth/INTEGRATION_NOTES.md`) — no email, no `analytics_events`/`ad_events` table, no Edge Functions at all exist yet (`supabase/functions/` is empty); (2) the one live serving path, `active_featured(surface, region)`, is real and safe (anon-callable `SECURITY DEFINER`, kill-switch + suspension checked server-side) but the *decision* of which ad wins is made client-side by `Explore.tsx:117`'s `featured[0]` — the RPC just orders by `starts_at asc` and ships up to 10 full creative rows to every browser.

**Top picks (3 lines).** Move ad selection into the RPC so rotation/fairness/creative-privacy are fixed in one place (#1); stand up the event pipeline that currently doesn't exist at all — nothing measures an impression or click today (#2); make the read path HTTP-cacheable since it's a public, PII-free `stable` function that currently pays a live 3-table join on every page load (#3). All three are additive, cost nothing new to run, and unblock most of the rest of this list.

---

## 1. Move ad selection server-side: rotate/select inside `active_featured`

Today `active_featured` (`supabase/migrations/20260712000000_growth_phase0_featured.sql`) returns up to 10 rows ordered by `starts_at asc`, and the only wired consumer (`Explore.tsx:117`) just renders `featured[0]`. Net effect: whichever campaign started first wins **forever** until it ends — a second paying advertiser on the same surface gets zero impressions for its whole flight; every anonymous visitor's browser also receives full creative text for every other active competitor on that surface even though 9 of 10 rows are silently discarded client-side, a real (if minor) competitive-intel leak for something advertisers pay for; and the sold `featured_capacity` inventory (`{"browse":3,...}`) has no relationship to what's actually shown. Fix: accept an anonymous `p_client_seed` (a random UUID the client mints once into `localStorage`, no PII) and pick the winner(s) inside the function via `order by hashtextextended(placement_id::text || p_client_seed, 0)`, returning only `p_limit` rows. This is the exact session-seeded shuffle `INAPP_ADS.md` §5.2 specified for the client — doing it in the RPC instead makes it authoritative and closes the creative-leak for free.
**Effort:** S/M. **Touches:** new migration delta on `active_featured`, `src/lib/api/growth.ts` (`activeFeatured` signature), `src/pages/Explore.tsx`. **Ship-first:** yes.

## 2. Minimal `ad_events` table + `record_ad_event` RPC

Confirmed by grep: no `analytics_events`, no `ad_events`, no `trackEvent`/`record_event`/`log_event` exists anywhere in applied migrations or `src/`. `FeaturedCard.tsx` fires nothing on click. Advertisers currently have **zero** way to know an ad was ever seen — the entire "event ingestion pipeline" domain is a blank slate. Don't wait for A4's full analytics platform: ship a narrow, dedicated `ad_events(id bigint generated always as identity, event_type text check (in ('impression','click')), placement_id uuid, campaign_id uuid, business_id uuid, surface text, session_id text, occurred_at timestamptz default now())`. Bigint identity instead of the house uuid-pk convention is a deliberate, flagged exception — this is the one genuinely high-volume append-only table in the schema, and every byte/row compounds. Writes go through a `SECURITY DEFINER record_ad_event(...)` RPC, never an open insert policy — unlike the (still-unbuilt) generic `analytics_events` design, ad events have real $ stakes for advertisers, so validation belongs in a function, not just a CHECK constraint.
**Effort:** S/M. **Touches:** new migration, `src/lib/api/growth.ts` (`recordAdEvent`), `src/components/growth/FeaturedCard.tsx`, new `useFeaturedImpression` hook (per `INAPP_ADS.md` §6.2 pattern, currently unbuilt). **Ship-first:** yes.

## 3. GET-cacheable `active_featured` with short `Cache-Control`

`supabase-js`'s `.rpc()` defaults to POST, which is never HTTP-cacheable — every `Explore.tsx` page load is a live Postgres round trip through 3 joined tables (`featured_placements` × `ad_campaigns` × `businesses`), with no caching layer below React Query's 5-minute client-side `staleTime`. `active_featured` is `stable`, read-only, and needs no auth — exactly the case `.rpc(name, args, { get: true })` exists for: it issues a GET, and PostgREST can emit `Cache-Control` a browser (or any CDN in front of the API in the future) will honor. Keep the TTL short — 30–60s — so `admin_set_campaign_status`/`admin_suspend_business` (`20260712030000_growth_admin_controls.sql`) still take effect fast; this directly serves "keeping serving fast on a free-tier Postgres" for close to zero engineering cost.
**Effort:** S. **Touches:** `src/lib/api/growth.ts` (`activeFeatured` → `{ get: true }`), migration (response headers via the `PGRST`-sqlstate header pattern already documented in `docs/ops/RATE_LIMITING.md`). **Ship-first:** yes.

## 4. Local/anonymous frequency capping, tradeoffs stated up front

Nothing caps featured-ad repetition per viewer today. The hard constraint rules out accounts, so capping has to be device-local: a small rolling log `{ [placementId]: [timestamp,...] }` in `localStorage`, capped to `INAPP_ADS.md`'s own numbers (≤3/placement/day, ≤10/surface/day, §5.3), consulted client-side to skip an otherwise-eligible placement and fall through to the next server-ranked candidate — which idea #1 makes possible by returning more than one ranked row instead of just the winner. Tradeoffs, stated rather than hidden: it's per-*device* not per-*person* (a phone+laptop pair effectively doubles the cap); it's trivially spoofable by clearing storage or private browsing; it undercounts more than it overcounts, which is the safe failure direction (`INAPP_ADS.md`'s own "fails user-friendly" principle) — acceptable because the downside of a miss is "saw an ad twice," not a security issue.
**Effort:** S/M. **Touches:** new `src/lib/ads/frequencyCap.ts`, `src/pages/Explore.tsx`. **Ship-first:** no — depends on #1 returning >1 candidate.

## 5. Reliable click capture via an Edge Function redirect

`FeaturedCard.tsx` renders a plain `<Link>`/`<a href>` today with no click handler at all. Even once #2 ships a `record_ad_event` RPC, firing it `onClick` races the browser's navigation and routinely gets dropped — especially on the `target="_blank"` external-link branch, or on a slow connection where the tab starts navigating before the fetch completes. The reliable pattern is a redirect endpoint: `/go/{token}` resolves a short-lived signed token to `(placement_id, destination_url)`, records the click server-side, then issues a 302 — captured *before* navigation can race it away. This would be the **first Edge Function the ads domain actually uses** (`supabase/functions/` is currently empty); it belongs there specifically because Postgres/PostgREST cannot emit a redirect — the one primitive here that structurally requires Deno rather than another RPC.
**Effort:** S/M. **Touches:** new `supabase/functions/go/index.ts`, `src/lib/api/growth.ts` (creative `link` → signed `/go/` URL), `src/components/growth/FeaturedCard.tsx`. **Ship-first:** no — #2's RPC-based click event is enough to ship the pipeline; add this once click volume/fraud matters.

## 6. Server-enforced slot capacity from `growth_settings`

`growth_settings.featured_capacity` (`{"browse":3,"map":1,"detail":1}`, seeded in the phase-0 migration) is never read by anything. `admin_review_campaign` inserts a `featured_placements` row on every approval with no capacity check, and `active_featured` hard-codes `limit 10`. Nothing stops a 4th concurrent `browse` placement from being approved — it would just silently underdeliver every advertiser on that surface once #1's rotation starts dividing impressions among 4 instead of the sold 3. Add a one-line guard in `admin_review_campaign` (count active placements for `(surface, region)` against the setting before insert) and make `active_featured`'s `limit` read the same setting instead of the hard-coded `10`.
**Effort:** S. **Touches:** migration delta on `admin_review_campaign` + `active_featured`. Cross-ref: `docs/growth/ABUSE_AND_LIMITS.md`'s anti-monopoly cap (R.7 in `DATA_MODEL.md`) is the fuller version of this; this is the cheap floor to ship before that. **Ship-first:** no.

## 7. Client-side coarse-region heuristic as a bridge (no Edge Function needed yet)

`active_featured` has real region-matching logic, but it's structurally inert: `Explore.tsx:48` calls `activeFeatured('browse')` with no region argument, so `p_region` is `null`, and the RPC's `(p_region is null or fp.region is null or fp.region = p_region)` clause matches *everything* regardless of targeting — every region-targeted campaign is effectively untargeted right now. `LOCATION.md`'s real fix (a MaxMind-backed `ip-geo` Edge Function) is a bigger build with a license/refresh-cadence decision attached. A zero-infra bridge that unblocks region targeting today: derive a coarse region client-side from `Intl.DateTimeFormat().resolvedOptions().timeZone` (a hand-maintained timezone→region lookup of a few dozen entries) — no network call, no consent question (it's a browser locale setting, not personal data), good enough for "which quarter of the country" granularity. Ship it as the default region source until the real geo pipeline lands; swap the source, not the plumbing.
**Effort:** S. **Touches:** new `src/lib/ads/coarseRegion.ts`, `src/pages/Explore.tsx`. **Ship-first:** no, but cheap enough to bundle with #1's rollout.

## 8. `ad_daily_stats` rollup + short raw retention via `pg_cron`

`SCALING_COST.md`'s central warning — `analytics_events` is the fastest-growing table by ~10× and breaks the free-tier 500MB around 1–2k DAU — applies to #2's `ad_events` too, just on a longer runway (ad-only traffic is a fraction of all-events traffic). Don't wait for A4's full rollup design: a narrow `ad_daily_stats(day, campaign_id, placement_id, surface, region, impressions, clicks)` table, refreshed nightly by a `pg_cron` job (same idiom as the already-shipped `dispatch_inapp_blasts` 5-minute schedule and `RATE_LIMITING.md`'s bucket-cleanup pattern), aggregates and then prunes `ad_events` rows older than 3–7 days — shorter than A4's proposed 7–30 day generic window, because ad reporting only ever needs daily granularity, never a specific timestamp. This is what should actually back `Campaigns.tsx`/`AdminCampaigns.tsx` reach numbers, so they never scan raw `ad_events`.
**Effort:** M. **Touches:** new migration (`ad_daily_stats` + `pg_cron` job), `src/pages/business/Campaigns.tsx`, `src/pages/admin/AdminCampaigns.tsx`. **Ship-first:** no.

## 9. First-party creative URL allowlist

`ad_campaigns.creative` is free-form jsonb (`{title, body, link}` today; `image_url` is designed but unbuilt). Nothing validates that `link`/`image_url` point at Watrloo-controlled infrastructure. Two reasons to gate this before images ship: (a) an `<img src>` to an arbitrary third-party URL is a tracking pixel — the viewer's IP hits an uncontrolled server with zero consent gating, exactly the "no third-party ad networks/trackers" self-sufficiency line the platform already draws for itself in `GROWTH_CONTRACT.md`, just not yet enforced against *advertiser-supplied* content; (b) only first-party (R2/watrloo.com) URLs can carry the long-lived immutable `Cache-Control` that makes creative assets cheap to serve (content-hashed filename, not query-string cache-busting, so R2 Class-B reads stay cacheable). Enforce as a check inside `submit_campaign`/`admin_review_campaign` (reject non-allowlisted hosts), not just a console-UI validation someone can bypass with a direct RPC call.
**Effort:** S. **Touches:** migration delta on `submit_campaign`. Cross-ref: `docs/growth/CAMPAIGNS.md`. **Ship-first:** no.

## 10. Client-side event batching + `sendBeacon` flush

Once #2's `record_ad_event` exists, naive per-event RPC calls (one insert per card mount, times however many surfaces a viewer scrolls past) mean one Postgres round trip per event — fine at today's volume (only `browse` is wired), not fine once `map`/`detail` get client integrations too. Queue events in memory; flush on a short interval (~2s) or on `visibilitychange`/`pagehide` via `navigator.sendBeacon` (survives tab close, unlike a `fetch` cancelled mid-navigation — the general-purpose cousin of #5's redirect fix, for impressions where a redirect doesn't apply). Batch multiple queued events into one RPC call taking an array so N impressions cost 1 Postgres call, not N.
**Effort:** S. **Touches:** new `src/lib/ads/eventQueue.ts`, `src/lib/api/growth.ts` (`recordAdEvents`, plural). **Ship-first:** no.

## 11. Event-integrity signature against anon-key event forgery

The anon key is public by design, which means anyone can call `record_ad_event` (#2) directly and manufacture impressions/clicks for or against any `campaign_id` — and campaign IDs aren't secret, they come straight back from `active_featured`. Since advertisers are billed/entitled against these numbers (`max_recipients_per_blast`, reach reporting), integrity matters more here than for generic analytics. Cheap mitigation: have `active_featured` return a short-lived HMAC (`hmac(placement_id || time_bucket, server_secret)`) alongside each row, and require `record_ad_event` to validate that token for the same placement/time-bucket before inserting. Doesn't stop a determined attacker (they can fetch-then-forge), but kills the "curl the RPC in a loop" class of attack for free — reusing the exact `check_rate_limit`-style server-secret pattern already established in `docs/ops/RATE_LIMITING.md`.
**Effort:** M. **Touches:** migration delta on `active_featured` + `record_ad_event`, a Vault secret for the HMAC key. **Ship-first:** no.

## 12. Unified `useActiveFeatured` hook across surfaces

Only `browse` is wired client-side; `map` and `detail` are fully built server-side (the surface enum, the RPC, `admin_review_campaign`'s placement insert path) but have zero consumers. When they're wired, don't repeat `Explore.tsx`'s inline `useQuery` shape three times over — factor a `useActiveFeatured(surface, region)` hook now, on one `queryKeys.featured(surface, region)` cache namespace, so #1's client-seed and #4's frequency-cap filtering live in exactly one place instead of drifting across three page components as each surface gets built out.
**Effort:** S. **Touches:** new `src/hooks/useActiveFeatured.ts`, `src/lib/queryClient.ts` (extend `queryKeys.featured` for region). **Ship-first:** no.

## 13. Materialized serving snapshot for scale-out beyond HTTP caching

#3's `Cache-Control` buys free repeat-reads within a TTL window, but every cache *miss* (first visitor after expiry, or a client that ignores `Cache-Control` on fetch/XHR — common in native webviews) still pays the full 3-table join with a `now() between` scan. If ad surfaces multiply (`browse`+`map`+`detail` all wired, #6's capacity raised) and this starts to show up in `EXPLAIN ANALYZE`, the next lever isn't a bigger Supabase plan — it's a `pg_cron` job every 30–60s that pre-computes the current winner set per `(surface, region)` into a tiny lookup table, and `active_featured` becomes an indexed point-read of that snapshot instead of a live join. Same shape as the already-shipped `dispatch_inapp_blasts` scheduler, just read-side instead of write-side. Explicitly a later-stage idea — the live-join RPC is already fast enough for a handful of advertisers; build this when the numbers say otherwise, not before.
**Effort:** M. **Touches:** new migration (`ad_serving_snapshot` + `pg_cron` refresh), `active_featured` rewrite. **Ship-first:** no — defer until traffic justifies it.

## 14. Documented partition/graduate glidepath for `ad_events`

Mirror `SCALING_COST.md`'s own methodology rather than inventing a new one: state, in #2's `ad_events` migration header, the exact trigger for graduating from "flat table + BRIN + 3–7 day retention" (#8) to monthly range-partitioning (the same mechanics as A4's `analytics_events` design, `DATA_MODEL.md` §R.9) — e.g. "when `ad_events` exceeds ~50k rows/day, or when Supabase Pro's idle-pause removal is needed anyway for another reason." Costs nothing to write down now and stops a future contributor from either partitioning prematurely (needless complexity for a handful of advertisers) or missing the point where a flat table starts to hurt.
**Effort:** S (a documented threshold, no code). **Touches:** `ad_events` migration header comment, or an addendum to `docs/growth/SCALING_COST.md`. **Ship-first:** no.
