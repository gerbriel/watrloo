# Reporting, Dashboards & Exports — Feature Ideas

**Top picks (build first):** CSV export off the existing `campaign_metrics` rollup is nearly free and closes the "I need this in a spreadsheet" gap immediately. Anomaly flags and date-range comparison both read the same `campaign_daily` rows already fetched for `CampaignMetrics` — they're UI-only diffs over data the console already has, not new pipelines. Together these three make the advertiser console feel like a real analytics product for a few hours of work, with zero new backend surface.

---

## 1. CSV export of campaign metrics

A "Download CSV" button on `CampaignDetail`/`CampaignMetrics` that serializes the `campaign_daily` rows already returned by `campaign_metrics(campaign_id)` (§7 of `ANALYTICS.md`) into a client-generated CSV (`Blob` + `<a download>`, no server round-trip). Columns: day, reach, impressions, opens, clicks, CTR, conversions — exactly what the advertiser already sees on screen, just portable. Because the RPC already applies the k-anonymity floor and returns aggregates only, the export inherits that privacy boundary for free; there is no separate export path to audit. Same pattern extends trivially to `EmailHealthPanel`'s admin suppression list and `analytics_daily` product rollups later.

**Effort:** S
**Touches:** `src/pages/business/CampaignMetrics.tsx` (or wherever A10's component lands), a small `toCsv()` helper in `src/lib/`. No new RPC, no new table.
**Ship-first:** yes

## 2. Anomaly flags ("CTR dropped 40%")

Compute simple week-over-week or day-over-day deltas client-side from `campaign_daily` (already the shape `CampaignMetrics` reads) and surface a banner/badge when a metric crosses a threshold (e.g. CTR down >30% vs the trailing 7-day average, or opens flatlining for 3+ days on a `running` campaign). No ML, no new table — just a `useMemo` over the existing series with a couple of documented thresholds (owned by this doc, tunable later). Render as a `Flag`-style pill (reusing `BusinessAnalytics`'s `warn` tone) with plain language, e.g. "Clicks are down 42% vs last week." This is the single highest-leverage feature for advertisers who don't have time to read a chart — it puts the insight in front of them instead of making them find it.

**Effort:** S/M
**Touches:** `CampaignMetrics` component (client-side delta calc only); optionally a shared `src/lib/analytics/anomaly.ts` helper so the same logic later serves the admin `EmailHealthPanel` (bounce-rate spikes) and `AdminAnalytics`.
**Ship-first:** yes

## 3. Date-range comparison (this period vs. last period)

A period picker (7/30/90-day presets, or custom range) on `CampaignMetrics` and the admin analytics dashboards that fetches two windows from `campaign_daily`/`analytics_daily` and renders "+18% reach vs. prior period" style deltas next to each KPI tile. Both rollup tables are already keyed by `day`, so this is a `where day between` re-query plus a second `Kpi` render pass — no schema change. Pairs naturally with idea #2 (the anomaly flag is really "comparison exceeded a threshold, shown automatically" — ship both off the same comparison utility).

**Effort:** S/M
**Touches:** `campaign_daily`/`analytics_daily` reads (existing), `CampaignMetrics.tsx`, `Kpi` component gets an optional `delta` prop, a `PeriodPicker` control shared with idea #11.
**Ship-first:** yes

## 4. In-app weekly recap ("bell" digest)

A lightweight in-app notification — not email — summarizing the past week per business: reach, CTR change, top anomaly if any, entitlement usage remaining. Generated by a small pg_cron job that writes a row per business into a new `business_digests` table (or reuses `moderation_actions`-style pattern), rendered as a dismissible card on `BusinessDashboard`/`CampaignList` ("Your week: 4,200 reached, CTR up 12%. View details →"). This is the pragmatic answer to "scheduled report emails" while Resend is still blocked on domain verification (`docs/ops/EMAIL.md` §0) — it delivers the same weekly-recap value with **zero email dependency**, and becomes redundant (or a nice belt-and-suspenders channel) once idea #7 ships.

**Effort:** S
**Touches:** new `business_digests` table + `roll_up_business_digest()` pg_cron job (mirrors `roll_up_analytics()`'s pattern in `ANALYTICS.md` §7.2); a `WeeklyRecapCard` component on the business dashboard.
**Ship-first:** no

## 5. Printable / PDF campaign report

A dedicated `/business/:businessId/campaigns/:campaignId/report` route with print-only CSS (`@media print`) that renders a clean, single-page summary — KPIs, the reach/CTR chart, the frequency-cap skip explanation — and relies on the browser's native "Print to PDF." Zero new dependencies (no `jspdf`/`html2canvas` bundle weight), fully consistent with "static SPA": the page *is* the report, the browser does the rendering. Good for advertisers who need to hand a PDF to a boss or client without screenshotting the dashboard.

**Effort:** S/M
**Touches:** new print-styled route/component reusing `CampaignMetrics` data; a `print.css` layer (or Tailwind `print:` utilities) hiding nav/chrome.
**Ship-first:** no

## 6. Shareable read-only report link

A "Share report" action on `CampaignDetail` that mints a signed, expiring token (`report_shares(campaign_id, token, expires_at, created_by)`) resolving to a **public, unauthenticated** read-only view at `/r/:token` — same aggregate `CampaignAggregate` shape as the advertiser sees, nothing more, no login wall. Useful for advertisers who want to forward results to a stakeholder who doesn't have a Watrloo account. The read path is a `SECURITY DEFINER` RPC (`get_shared_report(token)`) that checks expiry and re-applies the same k-anonymity floor as `campaign_metrics` — the share link can never expose anything the advertiser's own login couldn't already see. Default expiry 30 days; revocable from the console.

**Effort:** M
**Touches:** new `report_shares` table + RLS (no client SELECT; RPC-only), `get_shared_report(token)` RPC, a public route `/r/:token` outside `RequireAuth`, "Share/Revoke" UI on `CampaignDetail`.
**Ship-first:** no

## 7. Scheduled report emails (weekly/monthly digest via Resend)

The literal ask — a pg_cron job that renders each business's weekly/monthly performance into the branded email template (reusing A6's shared render function, per `ADVERTISER_CONSOLE.md` §5.5's request to A6) and sends it through the existing `campaign_sends`-adjacent Resend pipeline. **This is explicitly blocked today**: `docs/ops/EMAIL.md` §0 confirms Resend rejects every send except to the account's own address until a domain is verified (SPF/DKIM/DMARC). Spec and build the template/cron/opt-in toggle now so it's a flip-the-switch away once Stage B of `EMAIL.md` lands, but don't schedule it live — ship #4 (in-app digest) first as the interim channel, and reuse its aggregation logic here so the two never drift apart.

**Effort:** M
**Touches:** `send_business_digest()` Edge Function + pg_cron schedule, a `digest_opt_in` toggle in `BusinessSettings`, template shared with A6/A10's `EmailPreview`. Depends on `EMAIL.md` Stage B (domain verification) before going live — note prominently in the PR.
**Ship-first:** no

## 8. Admin revenue dashboard

A `/admin/analytics` (or new `/admin/revenue`) panel for MRR, plan-tier mix, upgrade/downgrade/churn counts, and revenue-by-plan over time — sourced from `subscriptions`/`plans` (already read by `BillingPlan`/`PlanPanel` per `ADVERTISER_CONSOLE.md` §5.8) rolled up into a small `revenue_daily` table by the same pg_cron pattern as `analytics_daily`. This is pure business-health reporting, not ad performance, but it's the dashboard admins will actually open daily once the paid tier has more than a handful of businesses. Charts follow the `dataviz` skill's palette/mark conventions, matching the existing `Kpi`/`ListingBar` visual language.

**Effort:** M
**Touches:** new `revenue_daily` rollup + `roll_up_revenue()` cron job, `admin_revenue_summary()` RPC (`is_admin()`-gated), a new admin panel slotting into `AdminAnalytics` from `ADMIN_CRM.md` §7.6.
**Ship-first:** no

## 9. Cross-campaign portfolio rollup

For a business running multiple concurrent campaigns (email + featured + newsletter), a "Portfolio" view above `CampaignList` that sums reach/CTR/conversions across all of a business's campaigns for a period — one query over `campaign_daily` grouped by `day` instead of `campaign_id`, no new table. Useful once a business has outgrown "look at one campaign at a time," and it's the natural home for the date-range comparison (#3) and anomaly flags (#2) at the account level rather than per-campaign.

**Effort:** S/M
**Touches:** `CampaignList.tsx` header section, a `campaign_daily` group-by-business query (existing table, new query shape), reuses `Kpi`/delta components from #3.
**Ship-first:** no

## 10. Admin rollup CSV/BI export

A generic "export this table" action on the admin analytics/email-health/revenue panels that dumps the currently-displayed rollup (`analytics_daily`, `campaign_daily`, `revenue_daily`, `email_suppressions` summary) as CSV for import into a spreadsheet or external BI tool. Same client-side `Blob` technique as #1, just pointed at admin-scoped RPCs instead of the advertiser-scoped one. Low effort because it's the same helper reused four times.

**Effort:** S
**Touches:** shared `toCsv()` helper from #1, "Export" buttons on `AdminAnalytics`, `EmailHealthPanel`, and #8's revenue panel.
**Ship-first:** no

## 11. Region reach chart

A coarse-region bar chart (per `dataviz` skill conventions — categorical palette, k-anonymity floor applied per bucket) on `AudienceExplorer` and `CampaignMetrics`, showing reach broken down by the region buckets already computed in `analytics_daily`/`campaign_daily`. Turns "top regions" from a SQL query (already sketched in `ANALYTICS.md` §7.3) into a visual an advertiser or admin can scan at a glance, rather than a table of numbers. Every bar respects the same `< 5` (advertiser) / `< 30` (segment) suppression floors already defined elsewhere — no new privacy surface, just a chart on existing aggregates.

**Effort:** M
**Touches:** a `RegionReachChart` component (client-side charting lib already in the app, or a lightweight SVG bar chart per `dataviz` skill), reads existing `campaign_daily`/`analytics_daily` region column.
**Ship-first:** no

## 12. Category/city benchmark ("your CTR: 24% · city average: 18%")

Show an advertiser how their campaign performs against an anonymized peer average — same city or same business category — computed from a new cross-business rollup (`benchmark_daily(region, category, day, avg_ctr, avg_open_rate, sample_size)`) that only ever exposes an aggregate over **N ≥ some floor (e.g. 10) distinct campaigns**, never a single competitor's number. This is the highest-value idea on the list for advertiser retention (context turns a raw CTR into a decision — "am I doing well?") but also the most sensitive to get right: it requires aggregating across *other businesses'* campaigns, which is a step beyond the "advertiser sees only their own campaign aggregates" boundary in `ANALYTICS.md` §6, so the floor needs to be strict and reviewed against that doc's k-anonymity posture before shipping.

**Effort:** L
**Touches:** new `benchmark_daily` rollup + cron job, `campaign_benchmark(region, category)` RPC with its own floor (separate from the existing per-campaign floor), a benchmark strip on `CampaignMetrics`. Needs a privacy sign-off pass (coordinate with the `ANALYTICS.md` owner) before build.
**Ship-first:** no

## 13. Campaign health score

A single letter grade or 0–100 score per running campaign (blended from CTR percentile, open-rate percentile, and frequency-cap skip rate) rendered as a compact badge on `CampaignList` rows and the top of `CampaignDetail`. Cheap to compute once #12's benchmark rollup exists (it's a percentile lookup against `benchmark_daily`); without it, a simpler self-relative version (score vs. this business's own historical average) ships standalone using only data already on hand. Mostly a retention/engagement device — gives advertisers a number to watch even when they don't want to read a chart.

**Effort:** S/M
**Touches:** a `computeHealthScore()` client-side helper (self-relative version needs no new backend); the full benchmark-relative version depends on #12.
**Ship-first:** no

## 14. Cohort funnel dashboard (admin)

A chart rendering of the signup → consent → first-review funnel SQL already written in `ANALYTICS.md` §7.3, currently just a raw query with no UI home. Wire it into `AdminAnalytics` (`ADMIN_CRM.md` §7.6) as a simple funnel/stacked-bar visualization per cohort week, so product/growth admins can see activation trending without running SQL by hand. Straightforward because the query already exists — this is "give it a chart," not "design a new metric."

**Effort:** M
**Touches:** `AdminAnalytics.tsx`, a `FunnelChart` component per `dataviz` skill conventions, existing SQL from `ANALYTICS.md` §7.3 wrapped in an RPC.
**Ship-first:** no

## 15. Annotated event markers on performance charts

Overlay vertical markers on the campaign performance chart at points where something changed the campaign — paused/resumed, creative edited, budget/frequency changed — sourced from the existing `moderation_actions`/campaign-status-transition audit trail (`ADMIN_CRM.md` §5.3, §2.3). Turns "CTR dipped on the 14th" into "CTR dipped on the 14th, right after the creative was swapped" — makes the anomaly flags (#2) actionable instead of just alarming. Purely additive UI over data that's already being written for audit purposes.

**Effort:** M
**Touches:** `CampaignMetrics` chart component, a read of the campaign's status-transition history (already audited per `ADMIN_CRM.md` §5.3), no new writes.
**Ship-first:** no
