# Privacy-First Measurement & Compliance — Feature Ideas

**Top picks (3 lines):** Ship the "Why am I seeing this ad?" transparency panel first — it's nearly free given the architecture is already contextual-only, and it converts the hardest constraint (no behavioral profiles) into the loudest marketing claim. Pair it with the self-serve Privacy Center (export + delete) to close COMPLIANCE.md's blocking DSAR gap before the first campaign ships. Third, unify the three different k-anonymity floors (5 / 25 / 30) already scattered across ANALYTICS.md, COMPLIANCE.md, and CAMPAIGNS.md into one enforced config value — it's a half-day fix that closes a real re-identification inconsistency before it becomes a launch-week bug report.

---

## 1. "Why am I seeing this ad?" transparency affordance

Every `FeaturedCard` / `FeaturedNearbyStrip` / `SponsorSlot` already carries a "Sponsored"/"Featured" label (INAPP_ADS.md §4) but never explains *how* it was picked. Add a small info affordance (an "i" glyph next to the disclosure pill) that opens a one-paragraph, plain-language explainer: *"Shown because you're browsing {region}. No profile of you was used — Watrloo never tracks you across sites, and advertisers never see who you are."* Because the targeting really is contextual (region + page only, per INAPP_ADS.md §3.2), this is a claim the system can make honestly with zero new data flows — it's UI + copy, reusing `AD_LABEL`/`AD_LABEL_LONG` from `src/lib/ads/disclosure.ts`. This is also the single highest-leverage move to turn the "contextual ads only" hard constraint into a differentiator against Yelp/Google, whose "why this ad" pages disclose profile-based targeting.
**Effort:** S. **Touches:** `src/components/ads/{FeaturedCard,FeaturedNearbyStrip,SponsorSlot}.tsx`, `src/lib/ads/disclosure.ts` (new `AD_EXPLAINER` copy + region interpolation), a shared `AdWhyModal`/popover component, `docs/growth/INAPP_ADS.md` §4 (add the explainer as a named requirement, not just a label). **Ship-first:** yes.

## 2. Self-serve Privacy Center — data export + account deletion UI

COMPLIANCE.md §8.2 flags this as blocking ("`REQUEST TO A2/A11`: an admin/self export RPC covering these tables") and PRIVACY_POLICY_v2.md §11 has a live `[DECISION NEEDED: confirm the self-serve account-deletion flow has been extended]` placeholder. Ship a Settings → Privacy & Communications page with two real actions: (a) **Download my data** — a `SECURITY DEFINER` RPC that assembles a JSON/CSV bundle of `profiles`, `user_consents`, `consent_events`, `user_locations`, `analytics_events` (own rows only), and `campaign_sends` received, delivered via signed download link (not email attachment, to avoid a new PII-in-transit surface); (b) **Delete my account** — extends the existing `delete_my_account` RPC (`src/lib/api/profiles.ts`) to purge/anonymize the new tables per COMPLIANCE.md §8.2, while preserving only the hashed `email_suppressions` row per §8.3. This is the single blocking item most likely to cause an actual regulator or user complaint if skipped, and it's the concrete mechanism DSAR handling requires.
**Effort:** M-L. **Touches:** new Edge Function or RPC `export_my_data(user_id)`, extension of `delete_my_account`, `src/pages/Settings/Privacy.tsx` (or equivalent), `docs/growth/COMPLIANCE.md` §8.2 (mark resolved), `docs/growth/DATA_MODEL.md` (A2 owns DDL/RLS for the export RPC). **Ship-first:** yes.

## 3. One shared k-anonymity floor (reconcile 5 / 25 / 30)

Three different suppression thresholds exist for what is conceptually the same guarantee: COMPLIANCE.md §9 says advertiser reach buckets suppress below "e.g. <25 users"; ANALYTICS.md §6's `campaign_metrics()` hard-codes `< 5`; CAMPAIGNS.md §2.3/§4.4 sets the reach-preview submission gate at `< 30` (`growth_settings.k_anon_floor`). Divergent floors across surfaces mean an advertiser could triangulate a narrow region from one aggregate view (floored at 5) that's blocked in another (floored at 30) — a real de-identification gap, not just an inconsistency nit. Fix: pick one number (recommend 30, the most conservative, already modeled as a `growth_settings` singleton in CAMPAIGNS.md §4.4), have `campaign_metrics()`, `campaign_estimated_reach()`, and any future admin-aggregate view all read the same `growth_settings.k_anon_floor` instead of hard-coding a literal, and add a lint/test that greps for stray numeric floors.
**Effort:** S. **Touches:** `growth_settings` table (already requested in CAMPAIGNS.md §4.4), `campaign_metrics()` in ANALYTICS.md §6, `campaign_estimated_reach()` in CAMPAIGNS.md §2.2-2.3, COMPLIANCE.md §9 (update the "e.g. <25" example to cite the real config value). **Ship-first:** yes.

## 4. First-party path for analytics/pixel/redirect endpoints (Safari ITP + tracker-blocklist resilience)

The `analytics-ingest`, `/e/o/<token>.gif` (open pixel), and `/e/c/<token>` (click redirect) endpoints in ANALYTICS.md §5.2/§9 currently resolve to `*.supabase.co/functions/v1/...` — a domain pattern that generic tracker-blocklists (Brave Shields, uBlock's default lists, some Safari content-blocker extensions) increasingly flag heuristically ("analytics", "supabase functions" showing up in EasyPrivacy-style lists), even though the data flow is entirely first-party and consent-gated. Fix by proxying these three paths under `watrloo.com/a/*` and `watrloo.com/e/*` (Cloudflare Worker or reverse-proxy rule, since Cloudflare already fronts the app per ARCHITECTURE.md) so requests are same-origin from the browser's and Safari ITP's perspective — same-eTLD+1 requests are exempt from ITP's cross-site tracking restrictions and are far less likely to hit heuristic blocklists keyed on third-party function-hosting domains. This directly serves the "must survive Safari/iOS" constraint and is cheap relative to the measurement fidelity it protects.
**Effort:** S. **Touches:** Cloudflare routing config (proxy rule), `src/lib/analytics/track.ts` (`VITE_SUPABASE_URL` → `VITE_ANALYTICS_BASE_URL`), `docs/growth/ANALYTICS.md` §5.2/§9.2-9.3 (document the proxy path), `docs/ops` (if a routing doc exists). **Ship-first:** no.

## 5. Consent receipt — user-facing history of their own `consent_events`

COMPLIANCE.md §1.3 requests an append-only `consent_events` audit table so the controller can "demonstrate consent" (GDPR Art. 7(1)) — but as designed it's an internal evidence record, not something the user themselves can see. Surface it: a read-only timeline in Settings → Privacy & Communications ("You turned on Marketing email on 2026-03-02 (Policy v2); you turned off Location on 2026-05-14") sourced from the user's own `consent_events` rows via a simple `select`-own RLS policy or thin RPC. This costs almost nothing beyond the table COMPLIANCE.md already asks for, and it's a rare case where the compliance artifact and a trust-building UX feature are the same data — "we can prove what you agreed to, and so can you" is a strong claim few competitors make visible to users.
**Effort:** S. **Touches:** `consent_events` RLS (own-row `select` policy, or a thin RPC if RLS is deemed too permissive for an audit table), Settings UI component, COMPLIANCE.md §1.3 (note the user-visible use case). **Ship-first:** no.

## 6. Config-driven, publicly-verifiable retention schedule

PRIVACY_POLICY_v2.md §11 has four separate `[DECISION NEEDED: confirm window]` placeholders for retention periods that COMPLIANCE.md §8.1 already proposes as defaults (13 months location, 14 months analytics, 24 months campaign sends, indefinite/hashed suppression). Risk: the policy text and the actual `pg_cron` purge jobs can silently drift apart over time (a policy is edited without the job being updated, or vice versa) with no mechanism to catch it. Fix: put the numbers in one `growth_settings`/`retention_policy` config table that both (a) the `pg_cron` purge jobs read for their `WHERE occurred_at < now() - interval` clauses, and (b) a public `/privacy/retention` page or Settings panel renders live, so the published promise and the enforced behavior are provably the same source of truth rather than two hand-maintained copies.
**Effort:** M. **Touches:** new `retention_policy` config table, the `pg_cron` jobs referenced in COMPLIANCE.md §8.1 / ANALYTICS.md §8, a small public-facing retention page, `docs/growth/DATA_MODEL.md` (A2 owns the table). **Ship-first:** no.

## 7. Policy-version change → targeted re-consent gate

COMPLIANCE.md §1.3 stamps every consent write with the `policy_version` in force at the time, and PRIVACY_POLICY_v2.md §19 promises material changes get "any legally required choices... before the change takes effect." Today that's a paper promise with no mechanism. Build a lightweight gate: when `CURRENT_POLICY_VERSION` bumps for a *material* change (new data use, new processor, retention change), any signed-in user whose latest `consent_events.policy_version` predates the bump sees a one-time, dismissible-only-after-reading interstitial ("We updated our privacy policy — here's what changed for you") on next login, logged as a new `consent_events` row with `source='policy_update_ack'`. Non-material edits (typo fixes) don't trigger it — a simple `is_material boolean` flag on a `policy_versions` table decides.
**Effort:** M. **Touches:** new `policy_versions` table, `AuthProvider.tsx` bootstrap check, an interstitial modal, `consent_events` schema (already exists per COMPLIANCE.md §1.3). **Ship-first:** no.

## 8. Public "Privacy & Ads" transparency page

Turn the compliance posture into a standalone marketing/trust asset instead of leaving it buried in a legal policy: a `/trust` or `/privacy/how-ads-work` page with a plain-English breakdown — "No third-party ad networks. No cross-site tracking. No cookies. Advertisers get counts, never names." — plus a live-updated badge pulling from the retention config (idea #6) and a link to the GPC-honoring statement. This is explicitly the "lean into contextual-only ads as a marketing advantage" instruction from the brief: most competitors (Yelp, Google) can't truthfully make this page, and a dedicated URL is link-bait for privacy-conscious press/communities (Hacker News, r/privacy) at effectively zero legal risk since every claim is already true by architecture.
**Effort:** S-M. **Touches:** new static route/page, content sourced from COMPLIANCE.md §6.2/§9 and PRIVACY_POLICY_v2.md §7-8, footer/nav link. **Ship-first:** no.

## 9. Persistent GPC/DNT status widget with one-tap "honor fully" action

COMPLIANCE.md §4 specs a banner for when GPC conflicts with an explicit opt-in, but only as a one-off notice. Make it a persistent, always-visible status line in Settings ("Your browser is sending a Global Privacy Control signal 🟢") with a single button that, when GPC is detected, offers "Turn off Location + Marketing to match my browser's signal" — collapsing COMPLIANCE.md §4 rule 2's "notify of conflict and ask them to confirm" into one tap instead of a passive banner a user might miss. This is cheap, directly closes the exact enforcement gap that cost Sephora $1.2M (COMPLIANCE.md §4, §11), and is good insurance regardless of how the "no sale/share" legal conclusion holds up over time.
**Effort:** S. **Touches:** Settings Privacy panel, existing `gpc_detected` read from `user_consents` (COMPLIANCE.md §4), a confirm-and-toggle-both RPC call. **Ship-first:** no.

## 10. Config-driven regional-rules matrix (`state_privacy_rules`)

COMPLIANCE.md §6.4 flags that VA/CO/etc. state laws each have their own thresholds and rights and will need periodic re-scanning; today every new state law would mean a policy rewrite and a code change. Instead, model a small `state_privacy_rules(region, right_key, label, url)` config table (e.g. `CA` → `gpc_honored`, `do_not_sell_link`; `VA` → `targeted_ad_optout`) that drives which rights/links render in the Settings "Your Rights" section based on the user's coarse region (already resolved via A3/edge geo, no new location processing). This turns "regional rules" from a doc-and-redeploy cycle into a data change an operator (or eventually counsel-reviewed ops runbook) can make without touching `src/`, and it scales cleanly if Watrloo ever expands beyond CA/US-federal-only compliance.
**Effort:** M-L. **Touches:** new `state_privacy_rules` config table, Settings "Your Rights" section (currently static text per PRIVACY_POLICY_v2.md §15), `docs/growth/COMPLIANCE.md` §6.4. **Ship-first:** no.

## 11. Click-first attribution UI + Apple MPP open-inflation labeling

ANALYTICS.md §9.5 already correctly diagnoses that Apple Mail Privacy Protection prefetches images and inflates `open_rate`, and says opens should be "indicative only" — but that's currently just a design note, not a shipped UI treatment. Concretely: (a) label `open_rate` in both the advertiser console and any admin dashboard with a visible "(includes automated prefetches, not verified opens)" tag rather than presenting it as a clean number; (b) default advertiser-facing campaign summaries to lead with clicks/conversions, with opens demoted to a secondary/expandable stat. This is a small UI decision that prevents advertisers from being misled by a metric the system already knows is unreliable — a measurement-integrity feature, not just a privacy one, but it's the direct, shippable follow-through on a gap the docs already identified and left as prose.
**Effort:** S. **Touches:** `src/pages/business/BusinessAnalytics.tsx` / advertiser console (A10), `campaign_metrics()` output labeling, `docs/growth/ANALYTICS.md` §9.5 (mark as implemented, not just diagnosed). **Ship-first:** no.

## 12. "Ads still show — here's why" expectation-setting copy on the consent toggles

Because ad serving is contextual and explicitly *not* gated on `marketing_opt_in`/`location_opt_in` (INAPP_ADS.md §3.2), a user who toggles both off in Settings will still see "Sponsored"/"Featured" cards — a mismatch against the intuitive (but wrong) expectation that privacy toggles reduce ads. Left unaddressed, this reads as a broken promise or a dark pattern on first discovery, which is corrosive to exactly the trust story this whole domain is built on. Fix with one line of microcopy directly under both toggles: *"This won't change which ads you see — we never use tracking to pick ads. It only controls whether we can email you and use your city for offers."* Cheap, and it pre-empts the single most likely user complaint about the ad system's honesty.
**Effort:** S. **Touches:** Settings → Privacy & Communications toggle copy (COMPLIANCE.md §1.2), possibly reused in idea #1's explainer modal. **Ship-first:** no.

## 13. Self-service "am I suppressed?" lookup tool

COMPLIANCE.md §8.3 retains a one-way-hashed `email_suppressions` record even after account deletion, specifically so a deleted/unsubscribed user is never re-mailed — but that also means a user who unsubscribed (especially pre-account or via the no-auth one-click endpoint) has no way to *verify* their own suppression status without emailing support. A tiny public tool at `/unsubscribe/status` — enter an email, server hashes it with the same function `email_suppressions` uses, returns "suppressed" / "not suppressed" (rate-limited, no other data revealed) — closes that loop and reduces support load, while being trivially safe since it discloses nothing beyond a boolean the user already knows the answer to (whether they clicked unsubscribe).
**Effort:** S. **Touches:** small Edge Function reusing the hash function from COMPLIANCE.md §8.3/§5.3, one static page, rate limiting (coordinate with `docs/ops/RATE_LIMITING.md` / A12). **Ship-first:** no.
